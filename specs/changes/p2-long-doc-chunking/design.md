# Design: p2-long-doc-chunking

## Summary
This change adds a semantic document-chunking capability and a new Doc2Doc service entry point so documents whose source-side token count exceeds the resolved `num_ctx` ceiling can be translated without truncation. Chunking logic lives in a new standalone module `doc_chunker.py` (owned area) that reads a parsed `TranslatableDocument`, groups its elements into `ChunkRecord`s at the highest-priority semantic boundary within the token budget (BR-50), inserts source-side overlap (BR-47), and reassembles per-chunk translations in `chunk_index` order with overlap de-duplication (data-shape Reassembly contract). A new `translate_document()` method on `translation_service.py` orchestrates split → translate-each-chunk → reassemble. The existing per-segment `translate_texts()` path, its cache key structure, and prompt templates are left untouched (BR-53). No HTTP surface, persistence, or IR wire schema changes; `ChunkRecord` is purely in-memory.

## Affected Components
| component | file path(s) | nature of change |
|---|---|---|
| doc chunker (new) | `app/backend/services/doc_chunker.py` | new module: `ChunkRecord` dataclass, boundary-priority splitter, overlap insertion, token estimator, reassembly + overlap de-dup |
| translation service | `app/backend/services/translation_service.py` | new `translate_document(doc, targets, src_lang, client, ...)` entry point; `translate_texts()` unchanged |
| config | `app/backend/config.py` | new `CHUNK_OVERLAP_TOKENS` constant (env-backed, default 50) |
| env contract artifacts | `contracts/env/.env.example.template`, `contracts/env/env.schema.json` | register `CHUNK_OVERLAP_TOKENS` (contract row already present) |

| orchestrator | `app/backend/processors/orchestrator.py` | no change this change; see Decision 4 |

## Key Decisions

### 1. Module boundary: standalone `doc_chunker.py`
**Chosen:** Keep chunking in a dedicated `doc_chunker.py` as the data-shape contract designates (it is the declared source of truth for `ChunkRecord`). `translation_service.translate_document()` owns orchestration and calls into the chunker; the chunker is pure (no LLM client, no I/O) so boundary/overlap/reassembly logic is unit-testable in isolation (test-plan AC-1, AC-2, AC-5). Rationale: keeps `translation_service.py` free of text-segmentation concerns and protects BR-53 — chunking cannot accidentally perturb `translate_texts()` shared state when it lives behind a separate module boundary.
**Rejected:** Embedding chunking inside `translation_service.py` — couples correctness-critical segmentation to the shared translation path, raises BR-53 regression risk, and makes the splitter harder to test without an LLM client.

### 2. Token counting: reuse the existing chars/token heuristic, no new tokenizer
**Chosen:** Use the same lightweight character-ratio estimator the codebase already relies on (orchestrator `_cap_terms_by_budget` uses `len(text) / 2.5`). The chunker estimates element token cost via a single shared `chars-per-token` divisor and compares against the resolved `num_ctx`. Rationale: P1 introduced no real tokenizer (confirmed — `num_ctx` is passed to Ollama as a window size, never used to count tokens locally); adding `tiktoken`/model-specific tokenizers would introduce a dependency and per-model accuracy coupling out of scope here. The ceiling check (BR-49) and overlap span (BR-47) are deliberately conservative so estimation error stays on the safe side of the window.
**Rejected:** Per-model exact tokenization — heavier dependency, model-coupling, and unnecessary for a conservative ceiling. The estimator divisor should be a named constant in `doc_chunker.py` so it is tunable and testable.

### 3. Overlap de-duplication: drop leading `overlap_element_count` elements from non-first chunks
**Chosen:** Per the data-shape Reassembly contract, on reassembly drop the translated output of the leading elements of every chunk with `chunk_index > 0`. Because elements are atomic and variable-length, `overlap_tokens` maps to `overlap_element_count` = the smallest number of leading elements whose combined estimated token span is ≥ `overlap_tokens`. The chunker records both the element set it duplicated as overlap and the resulting `overlap_element_count` on the `ChunkRecord` at split time, so reassembly drops *exactly the elements that were duplicated* rather than recomputing a token-span boundary against translated text (translated token counts differ from source). Rationale: anchoring de-dup on the element-count captured during split makes reassembly deterministic and avoids drift between source-side and target-side token estimates.
**Rejected:** Recomputing the overlap boundary from translated output token counts — target-language token counts diverge from source, so this risks dropping too few/too many elements (content duplication or loss, violating AC-5). Rejected: aligning on a shared text boundary string match — brittle across translation rewording.

### 4. Doc2Doc entry point: new `translate_document()`, not wired into orchestrator this change
**Chosen:** Add `translate_document(doc: TranslatableDocument, targets, src_lang, client, ...) -> TranslatableDocument` as a new public method that mutates and returns the input document in place (data-shape Doc2Doc contract). It internally builds chunks then calls the existing per-chunk translation primitive (e.g. `translate_blocks_batch`/`translate_texts` per chunk) — it does **not** wrap a single `translate_texts()` over the whole doc, because chunk isolation (AC-4: one LLM call per chunk) and BR-51 per-chunk failure handling require chunk-level boundaries. Orchestrator/processors are **not** modified: today the docx/pptx/xlsx processors call `translate_texts()` on deduplicated text lists and never construct a `TranslatableDocument` for the translation step (only the PDF parse/render path builds the IR). Wiring Doc2Doc into a processor is a separate downstream change; AC-7 only requires the entry point to chunk transparently when called, which it does.
**Rejected:** Making `translate_document()` a thin wrapper over one `translate_texts()` call — defeats the entire purpose (no per-chunk isolation, no ceiling enforcement). Rejected: modifying orchestrator now — no contract requires it, and it would expand blast radius into the shared translation flow without a consuming requirement.

### 5. Chunk failure handling: whole job fails, failed-chunk elements get BR-25 placeholder
**Chosen:** Per BR-51 and aligned with BR-7 job-lifecycle, a single chunk translation failure surfaces as an error and the job transitions to `status: "failed"` when no retry strategy is configured. The failed chunk's elements receive the BR-25 failure placeholder in `translated_content` (never null/silently dropped); reassembly of the remaining successfully-translated chunks is not corrupted. The implementation must explicitly record which chunks succeeded vs. failed so the content-integrity invariant holds. This matches the existing `translate_texts()` `fail_cnt` semantics rather than introducing a new partial-success state.
**Rejected:** Best-effort partial success that silently emits a partial document on chunk failure — contradicts BR-51's "job transitions to failed" and the content-integrity invariant.

### 6. Cache: Doc2Doc shares the existing cache key structure; no new variant
**Chosen:** Per-chunk translation goes through the same translation primitive and therefore the same cache key structure as `translate_texts()` — segment text + target + src_lang + model key. Doc2Doc MUST NOT introduce a new cache-key field or alter the existing one (BR-53 explicitly forbids changing the cache key structure `translate_texts()` depends on). Because chunking only changes how elements are grouped into LLM calls (not the per-segment cache unit), cache entries remain interoperable and no separate partition is needed. Rationale: minimal-surface, BR-53-safe, and lets long-doc translations reuse cached segments from prior runs.
**Rejected:** A separate Doc2Doc cache partition/variant — unnecessary (the cache unit is per-segment, unaffected by chunk grouping) and a maintenance liability. Rejected: disabling cache on the Doc2Doc path — wastes the existing per-segment cache for no correctness benefit.

## Migration / Rollback
No schema migration, no data migration, no persistence change — `ChunkRecord` is in-memory only and the `TranslatableDocument` wire schema is unchanged. The only environment delta is the new `CHUNK_OVERLAP_TOKENS` variable (default 50), which is inert on the single-chunk path and on all existing callers because `translate_document()` is not yet invoked by any processor. Rollout is additive: shipping the module and entry point changes no existing behavior. Rollback is removal of `doc_chunker.py`, the `translate_document()` method, and the `CHUNK_OVERLAP_TOKENS` config line; because no caller depends on the new entry point, removal is safe with no downstream breakage. The `CHUNK_OVERLAP_TOKENS ≥ num_ctx` misconfiguration is guarded by a `ValueError` at chunker init (data-shape Invalid-data table), failing fast rather than silently producing oversized chunks.

## Open Risks
- The chars/token estimator is approximate; if a real tokenizer is later required for tighter `num_ctx` packing, the divisor constant in `doc_chunker.py` is the single swap point. Track as future work, not a blocker at this tier.
- `overlap_element_count` derived from a coarse token estimate may occasionally drop one element more/fewer than an exact tokenizer would; mitigated by capturing the duplicated element set at split time so source/target estimates never disagree on reassembly. Data-boundary tests (oversized atomic segment, single-chunk-at-ceiling, mixed line endings) must cover the off-by-one element boundary.
- Doc2Doc is an unwired entry point after this change; a follow-up is needed before any format processor benefits from long-doc chunking. Flagged so it is not mistaken for an end-to-end-active feature.
