# Design: p2-text-expansion

## Summary
Replace the current "shrink to ~4pt then silently truncate" fit logic with an explicit, ordered fit cascade (font-size shrink → line-spacing → letter-spacing → controlled overflow → marked truncation) plus a metric-compatible font fallback chain. The fit cascade currently lives inline in `fitz_renderer.PDFGenerator._insert_text_in_rect` (the converged fitz primary path). It is extracted into a backend-neutral helper so the single converged path and the shared `bbox_reflow.py` placement layer drive one implementation (AC-6); the ReportLab fallback renderer reuses the same decisions. Font selection gains an x-height/cap-height/advance-width fallback heuristic in `font_utils.py` that defers to existing per-language Noto loading and reuses the P1 font-buffer LRU cache (AC-7). Truncation is last-resort and recorded on the IR element so the QA safety net can find it (AC-5).

## Key Decisions

**1. Cascade ordering and per-step thresholds.** Apply, in order, against the IR `bbox`: (a) **font-size shrink** from the language `FONT_SIZE_CONFIG.max` down to `min` (default 4pt) using `shrink_factor` per existing config; (b) if still overflowing at `min`, **line-spacing compression** from 1.15 down to a floor of 1.0; (c) **letter-spacing/tracking** reduction to a floor of -0.5% of em (negative tracking is capped to avoid glyph collision); (d) **controlled overflow** into adjacent vertical whitespace only — expand the text rect downward by up to a bounded delta (default ≤ 15% of bbox height) when the region below the bbox is whitespace per the placement sequence, never sideways into a neighbor's bbox; (e) **truncation** as the sole remaining option, clipping to the last whole word that fits and appending an ellipsis, always marked. Each step is attempted only when all earlier steps are exhausted, so no overflow occurs for en→de/es benchmarks before truncation (AC-1/AC-2/AC-4). The cascade returns a structured decision object; the fitz adapter and the ReportLab adapter both render from it. *Rejected:* a single combined "scale everything at once" solver — it produces non-reproducible results and cannot be asserted step-by-step in `test_text_region_renderer.py`.

**2. Metric-compatible fallback chain.** When `get_font_for_language` resolves a font that lacks a glyph for a target character, select a fallback by comparing candidate fonts' x-height, cap-height, and mean advance-width against the primary font's metrics and choosing the nearest (weighted: x-height, then cap-height, then advance-width). Candidates are drawn only from already-registered per-language Noto faces; **Noto (the language's `LANGUAGE_FONT_MAP` Noto face, else `NotoSans`) is the standard terminal fallback**. Selection reuses `register_fonts` registration state and loads bytes only through the P1 `_load_font_buffer` LRU cache, so fallback never triggers redundant font I/O (AC-7). Metric values are read once per registered face and memoized. *Rejected:* shipping a new bundled fallback font — violates AC-7 reuse and adds asset weight; rejected silently substituting Helvetica — produces tofu/metric drift, fails AC-3.

**3. Truncation marker representation.** The marker lives in the **IR**, as an additive optional field on `TranslatableElement` (e.g. `render_truncated: bool` defaulting `False`, set by the renderer when step (e) fires). It is a render-time *annotation* persisted on the IR element rather than a separate side-channel, so the QA safety net and human-review tooling read it through the same `to_dict()` surface they already consume, and a re-rendered IR carries its truncation history. This makes `data-shape-contract.md` an authoritative (not review-only) surface for this change. *Rejected:* a pure render-time log line — not machine-queryable, fails AC-5's "machine-readable" requirement; *rejected:* a new region-level `ElementType` — truncation is a per-element rendering outcome, not a structural type. See ADR 0004.

**4. Expansion-factor table and default policy.** A coefficient table covers en→de (1.30), en→es (1.25), en→fr (1.20) as the documented benchmark pairs (resolves AC-8). The factor is advisory: it pre-sizes the initial fit attempt (start the cascade assuming the translated string occupies factor×source length) so the cascade converges faster, but the measured rendered width always governs the actual decision — the table never overrides real metrics. **Any language pair not in the table defaults to a documented factor of 1.15** (a conservative Latin-expansion midpoint). The table and default live as a business rule in `business-rules.md`. *Rejected:* hard-coding factors as magic numbers in the renderer — not contract-governed, fails the AC-8 "documented" requirement.

## Migration / Rollback
No datastore migration, no schema/ENUM DDL, no env or secret change — all logic is in-process rendering and a font helper. The IR gains one additive optional field defaulting to `False`; old-format IR dicts lacking the key deserialize unchanged (matches the existing `reading_order` backward-compatibility rule), so serialized IR snapshots round-trip without loss. Rollback is a code revert: removing the cascade helper restores the prior inline shrink loop, and the additive IR field is ignored by any consumer that does not read it, so no data cleanup is required. The fitz/ReportLab primary-fallback contract (BR-34/BR-35) is unaffected; both paths simply consume the new shared cascade decision.

## Open Risks
- The code-map lists `pdf_generator.py` as the renderer, but the converged primary path is `fitz_renderer.py` (renamed in p2-renderer-convergence); `.cdd/code-map.yml` is stale on this point — recommend running `cdd-kit code-map` before implementation so the planner reads correct ranges.
- CER-001 (read access to `coordinate_renderer.py` / `inline_renderer.py` / legacy `pdf_generator.py`) is still `pending`; AC-6 "no duplicated expansion logic in legacy paths" cannot be fully verified by this review until that expansion is approved. Flagged for the planner/QA grep step.
- "Controlled overflow into adjacent whitespace" depends on placement neighbors; if `bbox_reflow` does not expose neighbor geometry, step (d) degrades to "no overflow, go straight to truncation." Implementation must confirm neighbor-whitespace availability or document the degraded behavior.
