# Design: p3-llm-judge

## Summary
Add an optional, feature-flagged LLM-as-judge quality review pass that runs after a translation job completes. A dedicated Gemma model (via Ollama, independent of the job's translation model) scores the collected source/translated block pairs on a 3-tier scale (‰Ωé/‰∏≠/È´ò). On ‰∏≠ or ‰Ωé the judge feedback is fed back into the existing translation path (`translation_service` / `OllamaClient`) for re-translation, capped at 3 attempts. The judge result (score, source text, translated text, feedback, attempt count) is attached to the in-memory `JobRecord` as a new optional field, exposed read-only via a new `GET /api/jobs/{id}/judge` endpoint mirroring the existing `/quality` pattern, and rendered in the frontend job detail panel. The judge is additive: when `JUDGE_ENABLED=false` or Gemma is unavailable, the job completes unchanged with no judge fields and no error.

## Key Decisions

**D1 ‚Äî Hook point.** The judge runs in `job_manager._run_job`, in the post-translate step, AFTER the QE scoring and term-audit blocks already there and BEFORE `status ‚Üí completed` ‚Äî it consumes the same `qe_blocks` accumulator the `post_translate_hook` already fills. No new orchestrator hook is added; this keeps COMET QE scoring the final translation and avoids a second pass through every processor (orphaned-wiring risk). The judge is sequenced after QE so QE still scores the originally rendered output. Re-translation calls reuse `translation_service`/`OllamaClient` directly on the collected block pairs ‚Äî never a parallel engine. Rejected: adding a new `judge_hook` callback threaded through all 5 processors (DOCX/PPTX/XLSX/PDF/DOC) ‚Äî multiplies wiring surface and re-render cost for no benefit, since the block accumulator already aggregates every format. Rejected: hooking before `post_translate_hook` ‚Äî would make COMET QE score pre-judge text, contradicting the requirement that QE runs on the final translation.

**D2 ‚Äî Iteration-state model.** Attempt count and per-attempt drafts are held in-memory inside the judge service for the duration of the post-translate step; only the final accepted result (winning score, final translated text, feedback, total attempt count) is written to `JobRecord.judge` once, at the end. This mirrors how `JobQualityRecord` is attached once on completion. Rejected: appending each attempt to the job record mid-loop ‚Äî would expose partial/non-terminal judge state to a polling client and complicate the read-only endpoint contract; the loop is short (‚â§3) and synchronous, so intermediate persistence buys nothing.

**D3 ‚Äî Job record data shape.** Add `JobRecord.judge: Optional[JudgeResult]`, strictly additive and parallel to `.quality`/`.audit` (data-shape-contract compatibility rule; existing `GET /jobs/{id}` consumers untouched). `JudgeResult` fields: `job_id`, `judge_status` (enum: available | disabled | unavailable), `score` (enum ‰Ωé|‰∏≠|È´ò, null unless available), `source_text`, `translated_text`, `feedback`, `attempts` (int), `model` (null unless available). The block-pair surface scored is the job-level `qe_blocks` aggregate; the recorded `source_text`/`translated_text` are a representative/joined view for display (per-cell judging is an explicit non-goal). `JobStatus` gains an optional summary `judge_score` for list views. Rejected: nesting judge data inside a generic `metadata` dict ‚Äî loses typing and diverges from the established typed `.quality`/`.audit` precedent.

**D4 ‚Äî Gemma routing.** The judge constructs its own `OllamaClient(model=JUDGE_MODEL)` against `OLLAMA_BASE_URL`, fully decoupled from the job's translation model/provider selection (which may be a cloud provider). `JUDGE_MODEL` defaults to `gemma3` and is a config/env value, not hardcoded; no new API key (local model). Rejected: routing the judge through `model_router`'s fallback chain ‚Äî the chain is for translation provider failover and could send judging to a cloud provider, violating the local-Gemma constraint and the no-DeepSeek-for-judging requirement.

**D5 ‚Äî Graceful degradation.** If `JUDGE_ENABLED=false`, judge step is skipped and `judge_status="disabled"`. If Gemma is unreachable / model not pulled / any judge exception, it is caught, logged at WARNING (never ERROR, never raised), `judge_status="unavailable"`, and the translation job completes normally with the originally rendered output delivered. This mirrors the QE `unavailable` contract exactly. The re-translation loop never blocks job completion: a failed re-translation attempt degrades to the last valid draft.

**D6 ‚Äî Score format.** The judge prompt elicits a single structured JSON object: `{"score": "‰Ωé|‰∏≠|È´ò", "feedback": "<reason>"}`. Score extraction parses JSON first; on parse failure it scans for the first of È´ò/‰∏≠/‰Ωé (exact token match ‚Äî synonyms are not inferred), defaulting to `unavailable` if none found rather than guessing. The normative prompt and the score-token contract are owned by business-rules.md; the `score` enum is the literal CJK tokens ‰Ωé/‰∏≠/È´ò to match user spec and UI copy. Re-translate trigger = score ‚àà {‰∏≠, ‰Ωé}; score È´ò stops immediately.

**D7 ‚Äî Re-render on confirm (apply path).** Re-render reuses the existing `processors.process_files` entry point against the ORIGINAL stored source file(s) under `job.input_dir`, writing back to the same `job.output_dir` filenames (`_output_name`), then re-runs `_archive_outputs` to rebuild `{job_id}_output.zip` and reassigns `job.output_zip`. The processors translate text via the LLM client ‚Äî they have no "inject pre-translated block map" seam today ‚Äî so feeding re-translated text back requires a per-block replacement surface, NOT a re-translation re-run (re-running would re-call the model and not reproduce the judged drafts). Therefore D7 requires storing the judge's accepted draft as a **per-block map** `{block_id: retranslated_text}`, not only the joined `judge.translated_text` (which is display-only per D3). The apply path passes that map through a new replacement hook the processors consult before calling the LLM (block_id present ‚Üí use stored text, skip the call). Implementation owns the exact hook signature; design constraint: the seam must be shared by all 5 processors via the same accumulator key (`block_id`) the `post_translate_hook` already emits, or the re-render will silently mismatch blocks. Overwrite is in-place (same output paths, same zip name) ‚Äî the download_url is stable; the destructive overwrite of the original translated file is per the change-request (no backup). Rollback is fail-soft: if any processor raises during re-render, the original `job.output_zip` is left untouched (rebuild into a temp path, swap only on success), `judge_apply_status="failed"`, and the user keeps the original download. Rejected: re-running translation from scratch with judge feedback in the prompt ‚Äî non-deterministic, would not reproduce the exact judged text the user confirmed, and doubles model cost. Rejected: rendering only the changed blocks into the existing output document in place ‚Äî the processors render whole documents, not block patches; partial-document mutation has no existing seam and risks layout corruption.

**D8 ‚Äî Apply endpoint.** Route `POST /api/jobs/{job_id}/judge/apply`, no auth (parity with `/cancel`, `/download`, `/judge`). Preconditions (else HTTP 409): `job.status == "completed"`, `JUDGE_ENABLED`, and a judge result whose `judge_status == "available"` with a non-empty per-block re-translated map (i.e. score was ‰∏≠/‰Ωé and a draft was produced); unknown job ‚Üí 404. The endpoint is **async**: re-render reuses the job worker model ‚Äî it sets `job.status` back to a transient `"applying"` (or a dedicated `judge_apply_status`) and dispatches the re-render on a daemon thread, returning `202` with `{"status": "applying"}`; the frontend polls the existing `GET /api/jobs/{id}` until terminal, then reads the (unchanged) `download_url`. Sync was rejected because re-render runs the full processor pipeline (whole-document render across N files) and can exceed request timeouts on large jobs, and because the codebase already standardizes long work on the background-thread + poll pattern (`useJobPolling`). Idempotency: re-render is deterministic given the stored per-block map (no model call ‚Äî D7), so a second apply rewrites identical bytes; a re-render already in progress (`status == "applying"`) returns `202` without dispatching a second thread (guarded by `job.lock`). Successful terminal response surface: `GET /api/jobs/{id}` reports `judge_apply_status="applied"` and the stable `download_url`; the response schema lives in `schemas.py` (`JobJudgeApplyResponse`) and is owned by api-contract.md.

## Migration / Rollback
No schema migration, no DDL, no persisted store change ‚Äî `JobRecord` is in-memory only and the new field is optional. Rollout is config-gated: shipping with `JUDGE_ENABLED=false` (the default) is a no-op; existing jobs, the `/quality` and `/audit` endpoints, and the `CRITIQUE_LOOP_ENABLED` path are unaffected (AC-8). Rollback is flipping `JUDGE_ENABLED=false` or reverting the change; no data cleanup is required because nothing is persisted. The new `/judge` endpoint returns `status:"disabled"` (HTTP 200) when the flag is off, so the frontend renders cleanly with the feature inert.

## Open Risks
- VRAM / model-load contention: Gemma loads alongside the local translation model on 8GB hardware; concurrent residency may thrash. Mitigation owned by implementation ‚Äî consider unloading the translation model before judging (the existing `release_resources` / `unload_model` path exists) and document it; flag stays off by default.
- Latency: up to 3 extra judge calls plus up to 3 re-translation passes per job add unbounded wall time on large jobs; the iteration cap (`JUDGE_MAX_ITERATIONS=3`) is the only guardrail and must be unit-tested (AC-3).
- Re-translation reflection (now in scope via D7/D8): the original render is written by each processor BEFORE the post-translate judge step; the downloadable document is updated only on explicit user confirm via `POST /judge/apply`, which re-renders from the original source. Until apply is confirmed, the download still serves the original output ‚Äî business-rules must state that the judge panel's improved text and the downloaded file diverge until apply.
- Re-render block-mismatch risk: D7's replacement seam keys on `block_id`. If the per-block re-translated map and the live processor block ordering ever diverge (e.g. a processor regenerates block ids on re-parse, or layout detection yields a different element set on the second pass), blocks silently mis-map and the re-rendered document is corrupted with text in wrong locations. Mitigation owned by implementation: assert id-set equality between the stored map and the processor's emitted block ids before substituting; on mismatch, fail-soft to the original download (per D7 rollback). Must be covered by a re-render integration test across all 4 formats.
- Re-render failure / fail-soft: re-render runs the full processor pipeline again and can raise (font, layout, COM/LibreOffice availability). D7 mandates temp-then-swap so the original `output_zip` survives; the risk is a half-written `output_dir` if the processor writes per-file before the rebuild completes ‚Äî implementation must rebuild into an isolated temp dir, not in place, or a crash mid-re-render leaves a mixed original/re-rendered output set.
- Apply idempotency / concurrency: two rapid apply calls, or an apply during a still-running re-render, must not spawn duplicate worker threads or interleave writes to the same `output_dir`. D8's `status == "applying"` guard under `job.lock` bounds this, but the 50-job-cap eviction and 24h TTL could evict the source/`input_dir` before a late apply ‚Äî apply must 409 if the original source is no longer on disk rather than re-rendering against missing input.
- Score-token ambiguity: small models may emit synonyms ("high"/"ËâØÂ•Ω") instead of È´ò/‰∏≠/‰Ωé; D6's strict-token parse plus `unavailable` fallback bounds this, but prompt quality is a tuning risk tracked in tests.
