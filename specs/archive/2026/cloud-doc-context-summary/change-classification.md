# Change Classification

## Change Types
- primary: feature-enhancement (enable an intentionally-skipped code path so the document-context summary runs on the active cloud translation client)
- secondary: business-logic-change (context-detection / `QWEN_CONTEXT_FLOW` gating now extends to the cloud path)

## Lane
- feature

Rationale: a deliberate enhancement of existing behavior — the guard
`... and _cloud_client is None` is removed and the summary call is parameterized
onto the active cloud client. Root cause and code location are already known and
named; this is not a symptom-driven bug. Bug-fix lane sections omitted.

## Risk Level
- medium

Changes the content of the translation system prompt on the user's real/primary
(cloud) path — every cloud translation now gets an extra summary LLM call plus an
injected preamble. No auth/payments/migration/data-shape/API-contract surface.
The explicit graceful-fallback requirement bounds the blast radius.

## Impact Radius
- module-level (backend document-context detection path: `orchestrator` + cloud
  LLM client + existing system-prompt injection wiring; no frontend, API route,
  DB, or cross-service impact)

## Inferred Acceptance Criteria
- AC-1: When translating via an active cloud provider (PANJIT/DeepSeek) with `CONTEXT_DETECTION_ENABLED` and `QWEN_CONTEXT_FLOW_ENABLED` on, the one-sentence document-context summary is generated using the ACTIVE cloud translation client, with no local Ollama model required.
- AC-2: The generated summary is injected as the document-context preamble ("Document context: <summary>") via the SAME orchestrator-side wiring the local path uses (`client.system_prompt` / `build_strategy`), AND the cloud translation client actually DELIVERS that `system_prompt` (scenario style + preamble) to the model as system-channel content on every `translate_once` call, merged ahead of the per-segment BR-78 `system_context` and never concatenated into the translatable user payload. Verified by asserting the summary reaches the outgoing request messages payload (not merely that the `system_prompt` attribute was assigned).
  - Note (scope correction, 2026-07-09): the cloud client previously carried `system_prompt` as an orchestrator-compatibility stub whose writes were "intentionally ignored", so neither scenario style nor summary ever reached a cloud model. A live capability probe of the PANJIT endpoint (`/v1/chat/completions`, gpt-oss:120b) confirmed system-role messages are fully honored (a system-only instruction overrode the user message verbatim), so the limitation was self-imposed on the client side and is removed by this change.
- AC-3: When either `CONTEXT_DETECTION_ENABLED` or `QWEN_CONTEXT_FLOW_ENABLED` is disabled, no summary is generated or injected on the cloud path (flags still gate the behavior).
- AC-4: When the active client is translation-dedicated (`client._is_translation_dedicated()` is true), summary generation is skipped exactly as before.
- AC-5: If the cloud summary call fails or returns empty, the translation job continues with no preamble (graceful degradation to current cloud behavior) and never aborts.
- AC-6: Local-Ollama context-detection behavior is unchanged — the summary is still generated and injected identically to before this change.
- AC-7: Scope is preserved: no JSON I/O change, no summary-prompt wording change, no flag rename, and no change to the ORCHESTRATOR-side injection wiring (`build_strategy` / the `client.system_prompt` assignment). The only new surfaces are (a) cloud summary generation via the shared `complete()` seam and (b) the cloud client honoring its own `system_prompt` in `translate_once`. The raw `complete()` summary call must carry NO system prompt. The per-segment BR-78 `system_context` behavior (step 1 / ADR-0016) stays intact: context is never concatenated into the translatable user payload.
