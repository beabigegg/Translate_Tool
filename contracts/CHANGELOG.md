## [business 0.34.0] — 2026-07-11
Added: BR-117 (`docx-cell-truncation-length-guard`). gpt-oss:120b returned 370 chars for a 4,827-char DOCX table cell with `ok=True` — schema-valid, coordinate-complete, non-echo, so both the JSON envelope (`table_serializer.parse_json`) and the legacy pipe-grid accepted it and over 90% of the content silently vanished. A new composition-aware length guard at the DOCX table-cell acceptance seam (docx_processor.py's existing BR-82 fallback block, ~L1088-1132) computes an expected-length model `E = a*cjk + b*latin_alpha` from a per-target coefficient table, seeded with Vietnamese `a=3.51, b=0.75` from calibration over 233 real cache pairs (0% false positives at every tested `k` from 0.2 to 0.5; the recorded bug sits at ratio 0.077), and flags `translated_len < k*E` at `k=0.3`. `k`, the coefficient table, and `MIN_SOURCE_CHARS=15` are `config.py` constants — NOT env vars, mirroring `MAX_TABLE_NESTING_DEPTH`/BR-113 and `MAX_GROUP_NESTING_DEPTH`/BR-116. The guard is fail-safe by construction: it never flags an uncalibrated target, a source below `MIN_SOURCE_CHARS`, or a zero-composition (all-numeric, BR-68-exempt) cell. On a flag, the cell is routed into the EXISTING BR-82 split-and-retranslate fallback, bounded to one non-re-entrant attempt (cannot loop); on exhaustion the LONGEST of {original reply, recovered reassembly} is kept and a `TranslateTool` WARNING is emitted — the guard NEVER substitutes source text and NEVER applies the BR-25 failure-placeholder format. The WARNING is observability-only with no automated consumer (`tests/metrics/truncation_rate.py` is unrelated — it counts render-time `render_truncated`, ADR-0004/BR-38, a distinct concept). No IR field is added or repurposed; `data-shape-contract.md` is untouched. Scope: DOCX table-cell acceptance seam only — the body/segment path and PPTX/XLSX table cells sharing `parse_json` are explicitly out of scope, follow-up on evidence. See ADR-0020. Added in change `truncation-length-guard`.

## [business 0.33.1] — 2026-07-11
Fixed: BR-115 (`header-footer-native-collection`) — the rule's scope, previously limited to header/footer, is extended to the body and table-cell paragraph extraction it had explicitly left out. `_process_container_content(doc._body, "Body", 1)` now threads the existing `_p_text_no_txbx` extractor (added for header/footer by this same rule) instead of the default `_p_text_with_breaks`, so a body paragraph or table cell hosting a `<w:txbxContent>` textbox no longer folds the textbox's text into its own segment — the textbox's text is collected and translated exactly once via the dedicated `_txbx_iter_texts` path and restored only into the textbox. The restore-time existing-translation (resume-idempotency) checks inside `_insert_docx_translations` — the SDT-content branch, the table-cell branch, and the plain-paragraph tail-scan (`_scan_our_tail_texts`) — are switched to the same `_p_text_no_txbx` extractor for family consistency with collection; all three reads only ever see paragraphs the pipeline itself previously inserted (plain runs, no nested textbox), so this is inert to their result but keeps one extractor family across the whole txbxContent-strip mechanism. `_txbx_iter_texts`'s own internal paragraph-text extraction (its private `_p_text_flags` helper, not `_p_text_with_breaks`) is unchanged; it remains the sole collection path for hosted-textbox text. No golden regression fixture contains a body textbox, so textbox-free DOCX output is unaffected and required no re-baseline. No env, API, or data-shape change. Fixed in change `docx-body-textbox-dedup`.

## [business 0.33.0] — 2026-07-11
Added: BR-116 (`pptx-group-shape-collection`). The PPTX collection walk only ever iterated `slide.shapes` flatly; a `GroupShape` reports `has_table=False` and `has_text_frame=False`, so every text frame and table nested inside a group was silently skipped and never translated. The walk now additively recurses into `GroupShape.shapes` (grouped text frames AND grouped tables), bounded by a new hardcoded `MAX_GROUP_NESTING_DEPTH` `config.py` constant (NOT an env var, mirroring `MAX_TABLE_NESTING_DEPTH`/BR-113 and `CONTEXT_DETECTION_ENABLED`/BR-109) that still collects a group's contents rather than dropping them at the limit, logging one WARNING per over-limit group. Same loop: the table-cell grouping key `shape_id = id(shape)` — an lxml-backed python-pptx proxy address — is replaced by a per-presentation document-order counter, the SAME mechanism BR-113 already established for DOCX `table_id` and the SAME `id()`-recycling hazard BR-81 forbids relying on. As with DOCX, the `id()` key is MASKED in the live loop (each emitted cell segment retains its parent shape via `_Cell._parent`, so live tables cannot share an address); verified by sabotage — reverting the counter to `id(shape)` leaves all group tests green. The counter is required because that correctness rests on an unstated retention invariant this repo forbids relying on and because group recursion adds more shapes, NOT because the real loop currently collides (an isolated no-retention probe does: 30 shapes → 2 keys under GC). See specs/changes/pptx-group-shape-collection/evidence/id-key-masking.md. SmartArt (`_extract_smartart_texts`) is unchanged and out of scope. Existing flat-shape PPTX collection, translation, and write-back are unchanged. No env, API, or data-shape change. Added in change `pptx-group-shape-collection`.

## [business 0.32.0] — 2026-07-11
Added: BR-115 (`header-footer-native-collection`). DOCX header/footer paragraph text and tables (including nested tables) were dropped on every OS: the native collector walked only `doc._body`, and the only header-touching code, the Windows-only COM postprocess (`postprocess_docx_shapes_with_word`), translates header-anchored SHAPES only — a header `<w:tbl>` or paragraph is not a `Shape`, so 15-cell header tables were never translated on Windows either. The native path now collects, translates, and restores header/footer paragraphs and tables (via the existing BR-113/BR-114 `_process_container_content`/`_process_table` walkers, zero new table/restore code) across all six per-section `<w:hdr>`/`<w:ftr>` slots on both OSes. Shared/linked parts (`is_linked_to_previous`) are deduped by holding the `<w:hdr>`/`<w:ftr>` element itself in a set, never `id()` (BR-81/BR-113 lesson: CPython recycles a freed lxml proxy's address). The classification's assumed COM-vs-native mutual-exclusion invariant was refuted against live source (ADR-0019): the two paths own disjoint content domains (native = header/footer paragraph text + tables; COM = header-anchored shapes only), so exactly-once holds by construction, not by a switch, and the COM call site (`include_headers=True`) is unchanged. This is load-bearing on stripping `<w:txbxContent>` text out of the native header/footer paragraph extraction (a filtered variant of `_p_text_with_breaks`, which otherwise folds a hosted textbox's text into its paragraph and would double-translate it against the COM shapes pass on Windows); header-anchored textboxes remain COM-owned on Windows and unhandled on Linux, a pre-existing textbox-scope gap not worsened here. Body/table collection and the pre-existing body textbox double-count are unaffected. No env, API, or data-shape change — reuses the existing IR and walkers with no new field. See ADR-0019. Added in change `docx-header-footer-collection`.

## [business 0.31.0] — 2026-07-10
Added: BR-113 (`nested-table-collection`) and BR-114 (`layout-frame-cell-reroute`). The DOCX `<w:tbl>` walk now additively recurses into `cell.tables`, which it never read before — silently dropping 65 of 275 and 218 of 523 body paragraphs (7.3% and 25.2% of whitespace-normalized characters) on two real documents, both of which now collect 100%. The merged layout-frame cell was additionally emitted, and translated, once per spanned column: redundant characters fall from 26,203 to 101 on one document, and the largest single table cell handed to the LLM falls from 8,729 characters to 207. Evidence: `specs/changes/docx-nested-table-collection/evidence/real-document-coverage.md`. Each nested `<w:tbl>` is emitted as its own table group under its own `table_id` — a monotonically increasing per-document counter, deliberately NOT `id(<w:tbl>)` — with a private 0-based `(row, col)` coordinate space, bounded by a new hardcoded `MAX_TABLE_NESTING_DEPTH = 3` `config.py` constant (NOT an env var, mirroring BR-109/`CONTEXT_DETECTION_ENABLED`) that flattens-and-warns rather than drops at the limit. A cell's direct paragraphs are rerouted to the body path only under a structural conjunction — contains at least one nested table AND spans the full row width — defaulting to no-reroute whenever the two signals disagree; paragraph count is explicitly excluded as a signal, because a genuine data cell may legitimately hold many paragraphs and rerouting it would strip its row context. Amended: BR-81 (`table-cell-column-dedup`) gains one clarifying sentence — a merged `<w:tc>` spanning multiple columns is deduplicated by `<w:tc>` element identity before the `(tgt, src_text, col)` key is formed, so it is emitted and translated exactly once at its origin column instead of once per spanned column; the key's axis and shape are unchanged. The rule explicitly forbids keying that dedup on `id(cell._tc)`. Measured during implementation: CPython recycles the address of a freed lxml proxy, so a walk that records `id(cell._tc)` without retaining the element sees 8 distinct keys for a 60x5 table's 300 cells, against 300 when the elements are held. A first draft of this rule claimed an `id()`-keyed set would have turned the 17% silent drop into a 95% one; that was an overclaim and was corrected before merge. Sabotage-tested both ways: switching the main cell loop's set to `id()` keys leaves all 13 new tests green (every cell is emitted into a `Segment` before the next lookup, so its element stays live), while switching the depth-limit flatten path — which retains nothing — turns exactly one test red. The prohibition stands not because the main loop breaks but because its correctness rests on an unstated, untested retention invariant that a sibling path in the same module does not satisfy. For the same reason `table_id` moves off `id(<w:tbl>)` onto a document-order counter and `seen_par_keys` moves off `id(p._p)` onto the paragraph elements; both were correct, neither for a stated reason. `docx_processor.py` now contains no `id()`-keyed collection. Evidence: `specs/changes/docx-nested-table-collection/evidence/id-key-hazard.md`. Table T gains two companion rows (merged-cell single emit; the pre-existing cross-table `(src_text, col)` collision, flagged so it is not mistaken for new behavior). See ADR-0018. Added in change `docx-nested-table-collection`.

## [business 0.30.0] — 2026-07-10

BR-107 and BR-68 passthrough are unchanged. Added in change `json-structured-translation-io`.
